# 合规与研究伦理说明

## 一、公开数据边界

本项目只处理**公开可获取的社交媒体讨论材料**。研究主线不包含：

- 私信
- 封闭群组
- 非公开主页
- 受限内容
- 任何需要绕过平台机制才能获取的内容

仓库中的历史自动采集系统已归档，仅作为历史运行基础设施保留；当前活跃主线不再以高风险采集作为默认入口。

## 二、平台规则意识

本研究强调：

- 不提供绕过登录、验证码、风控、限流、设备指纹、封禁机制的方案
- 不把平台技术限制视作必须突破的工程目标
- 对于不可访问、已排队未处理或媒体尾部缺口，优先作为研究限制说明，而不是无限扩张采集行为

## 三、数据最小化原则

研究主库只保留分析所需的最小字段，重点关注：

- 帖子文本
- 评论文本
- 时间信息
- 查询词与来源说明
- 工作流、规范评价与边界生成相关标签

不在研究主库中保留与研究问题无直接关系的冗余个人信息。

## 四、匿名化规范

### 1. 用户标识

- 作者 ID 尽量哈希化处理
- 评论者 ID 尽量哈希化处理

### 2. 用户名称

- 作者名称尽量掩码化
- 评论者名称尽量掩码化或不进入主库

### 3. 身份敏感信息

- 不主动保存敏感身份信息
- 对 legacy 数据中的身份线索，只保留与研究相关的角色类别（如教师、研究生、机构、工具方）而不保留可直接识别个人身份的详细字段

## 五、引用与展示规范

在论文、汇报或图表展示中：

- 不直接暴露真实用户 ID
- 不直接暴露完整用户名
- 引用帖子和评论时，以去标识化摘录为主
- 如确需展示原始语句，应避免附带可直接定位个体的组合信息

## 六、风险与缓解措施

### 1. 数据风险

- 风险：尾部帖子或媒体内容不可访问、采集不完整
- 缓解：在论文中明确说明限制，不把尾部缺口包装为完整覆盖

### 2. 身份风险

- 风险：帖子或评论文本可能包含自我暴露信息
- 缓解：研究主库最小化字段、哈希化 ID、掩码化名称、摘录时进一步去标识

### 3. 平台风险

- 风险：历史运行链与平台限制高度相关
- 缓解：本轮重构后将其归档，活跃主线转向合规导入、研究型数据库和分析流程

### 4. 方法风险

- 风险：历史规则编码与结构修补可能影响样本边界
- 缓解：将边界收紧逻辑显式写入方法说明，并保留可追溯的 review / override 记录

## 七、当前边界说明

当前投稿主稿采用 `quality_v6 post-only formalized result layer`：

- 当前投稿正式帖子 / 正式评论：`714 / 0`
- 来源组合：`quality_v5 514` + `supplemental_formalization_v1 200`
- `quality_v5 post-only 514 / 0` 仍作为工程 guard 和前一冻结基线保留
- `comment_review_v2 deferred`
- `formal_comments=0` 是当前投稿口径的设计选择，不是导入遗漏
- `quality_v4` 仅作为 historical audit，不作为当前正式结果

历史审计中出现的排队、不可访问、媒体缺口和评论层材料，在当前阶段都被视为**残余限制或后续独立工作流**，而不是继续扩张采集行为的理由。研究主线优先聚焦编码分析、framework v2 支持层和论文写作材料。

如需继续扩展样本，当前仅允许以 supplemental / candidate 工件形式先行保存公开可获取帖子材料及其评论 sidecar 状态；候选集不得自动进入 `quality_v6` formal result layer，也不得通过绕过登录、验证码、风控、限流或封禁机制来补齐不可访问内容。任何补充材料进入正式结果层，都必须经过单独 formalization、source contract、consistency report 和 paper materials 更新。
