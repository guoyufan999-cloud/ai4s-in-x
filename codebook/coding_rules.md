# 编码规则说明（framework v2）

## 一、当前活跃口径

当前活跃框架不再是旧的“科研工作流—合法性评价—边界协商”，而是：

话语情境 -> 实践位置 -> 介入方式 -> 规范评价 -> 边界生成

旧 `A. 科研工作流环节识别`、`B. 合法性评价识别`、`C. 评价标准`、`D. 边界协商机制识别` 不删除；它们作为 historical audit、legacy mapping 和旧 artifacts 解释层保留。其中 A 组改释为实践位置，B/C 组改释为规范评价，D 组改释为边界类型与表达方式。

## 二、编码顺序

### 1. 先判话语情境

判断文本属于工具推荐、经验分享、教程展示、风险提醒、伦理批评、规范解读、评论争论或其他。优先使用 `discursive_mode`、`actor_type`、`speaker_position_claimed`、`record_type`、`context_used` 等既有字段映射。

### 2. 再判实践位置

判断 AI 被放进科研生产、科研治理、科研训练与能力建构中的哪个位置。A 组不再只称为工作流，而是实践位置。

### 3. 再判 AI 介入方式

使用 F 组判断 AI 做了什么：

- 信息辅助
- 生成辅助
- 分析建模
- 判断建议
- 自动执行
- 治理监督

### 4. 再判 AI 介入强度

使用 G 组判断强度：

- 低强度辅助
- 中强度共创
- 高强度替代

介入强度不是由工具名称决定，而是由具体使用方式决定。同样是 ChatGPT，用于翻译文献可能是 `G1`，用于生成论文核心论证可能是 `G3`。

### 5. 再判规范评价

使用 B 组判断规范评价倾向，使用 C 组判断规范评价标准，必要时使用 H 组记录评价张力、I 组记录正式规范参照。

### 6. 最后判边界生成

使用 D 组判断边界类型，使用 J 组判断边界协商机制，使用 K 组判断边界协商结果。

## 三、纳入与剔除

### 纳入标准

- 明确涉及 AI 或具体 AI 工具
- 明确回到科研实践、科研治理、科研传播或科研训练语境
- 至少能支持五层中的一个实质判断

### 默认剔除

- 泛化趋势帖、科技新闻帖、纯产品介绍、纯广告、纯链接
- 普通学习、办公、求职、一般编程，且无法回到科研实践位置
- 只有情绪表达、段子、口号或正文信息过薄，无法支持任何五层判断

## 四、五项自检

1. 话语情境是否能从文本表达方式或已有字段中判断。
2. 实践位置是否能回到科研活动。
3. 介入方式和介入强度是否描述 AI 实际做了什么、做到什么程度。
4. 规范评价是否有原文证据，是否区分倾向、标准、张力和正式规范参照。
5. 边界生成是否真的提出规则、限制、门槛、机制或结果。

## 五、结构化输出

framework v2 的人工编码可理解为以下信息块：

1. 是否纳入
2. 纳入或剔除理由
3. 主题摘要
4. 话语情境
5. 实践位置
6. AI介入方式
7. AI介入强度
8. 规范评价倾向
9. 规范评价标准
10. 评价张力
11. 正式规范参照
12. 边界类型
13. 边界协商机制
14. 边界协商结果
15. 备注

## 六、工程约束

- 新增 v2 字段通过 canonical JSONL、`reviewed_records.payload_json` 和 `claim_units` 承载。
- 不做数据库 schema migration。
- 未编码时允许为空、空数组或 null。
- 不允许程序自动推断这些字段。
- framework v2 人工补码使用 `framework_v2_update=true` 的 `post_review_v2` 修订批次；导入时不得改动既有 A/B/C/D 字段。
- DeepSeek/LLM 预填如果涉及这些字段，只能生成 draft，不能直接写入正式编码。
- 人工 reviewed import 才能形成正式结果。
- `comment_review_v2` 仍保持 deferred，不把 comment corpus 写成当前正式评论结果。
